# MT2004 购买数字 —— 思路对比笔记

## 一、题目关键信息（我之前可能忽略的）

| 限制项 | 具体内容 | 对我的影响 |
|--------|---------|-----------|
| `n` 的范围 | `1 ≤ n ≤ 10^200` | **200位数字！** 完全超出 `int` / `long long` 范围 |
| 输入格式 | 可能有**前导零**（如 `01452`） | 用 `%d` 读入会丢失前导零，且溢出 |
| 时间限制 | 1秒 | 枚举 200 位数从大到小数到宇宙热寂也数不完 |

> **致命误区**：我看到 "一个正整数 n" 就本能地 `scanf("%d", &n)`，这是做惯了小数字题的条件反射。

---

## 二、我的思路：枚举法（从 n 往下找第一个回文数）

### 逻辑流程
```
输入 n
for i 从 n 递减到 1:
    如果 i 是回文数:
        输出 i，结束
```

### 这个思路的问题

| 问题 | 说明 |
|------|------|
| **数据类型爆炸** | `n` 有 200 位，`int` 最多存 10 位（约 2×10⁹），`long long` 也只有 19 位 |
| **枚举不可行** | 即使 `n = 10^200`，从 `n` 递减到 `1` 需要枚举 `10^200` 次，1 秒内连 `10^9` 都跑不完 |
| **前导零丢失** | `scanf("%d")` 遇到 `01452` 会直接存成 `1452`，但题目要求保留前导零的语义 |
| **字符串转换麻烦** | 每次 `i--` 都要 `sprintf`，200 位数字的字符串操作极其频繁 |

### 枚举法真的错了吗？

**不是思路错，是用错了场景。**
- 如果 `n ≤ 10^6`，枚举法是完全正确的，代码简单直接。
- 但 `n ≤ 10^200` 时，枚举法从**算法复杂度**和**数据表示**两个层面都不可行。

---

## 三、正确思路：字符串镜像构造法

核心思想：**不枚举，直接构造**最大的、不超过 `n` 的回文数。

### 步骤拆解

```
输入：字符串 s（如 "1452"）

Step 1: 去前导零（保留至少一位）
        "01452" → "1452"

Step 2: 取前半部分，镜像到后半部分，得到候选回文 t
        "1452" → 前半 "14" → 镜像 "1441"

Step 3: 比较候选 t 和原数 s
        "1441" ≤ "1452"？ 是 → 直接输出 "1441"

        如果 t > s（如 s="110", t="111"）：
            Step 4: 前半部分减 1
            "11" → "10"

            Step 5: 重新镜像
            "10" + 镜像 → "101"

            Step 6: 如果减 1 出现前导零（如 "10"→"09"）：
                    说明同长度回文不存在，输出 m-1 个 9
                    "1000" → "999"
```

### 为什么这个方法可行？

| 对比项 | 枚举法 | 镜像构造法 |
|--------|--------|-----------|
| **时间复杂度** | O(10^200) —— 不可行 | O(L) = O(200) —— 常数级 |
| **空间复杂度** | 需要不断转字符串 | O(L) —— 固定数组 |
| **核心操作** | 递减 + 判断 | 字符串镜像 + 大数减 1（手动借位） |
| **适用场景** | 小数字（≤ 10⁶） | 任意长度（只要数组开得够） |

## 四、关键算法对比：以 "110" 为例

### 我的枚举法（理想化，假设能存大数）
```
110 → 109 → 108 → ... → 101（找到回文）
共检查 10 个数
```

### 正确构造法
```
s = "110"
前半 = "11"（含中间位）
镜像得 "111"
"111" > "110"？ 是，需要减 1

前半 "11" 减 1 → "10"
重新镜像 "10" → "101"
输出 "101"

只操作了 2 次字符串处理！
```

## 五、C 语言实现的痛点对比

| 操作 | Python（我熟悉的） | C 语言（我感到便秘的） |
|------|-------------------|---------------------|
| 读入大数 | `s = input()` 直接得字符串 | `char s[205]; scanf("%s", s);` 需预开数组 |
| 去前导零 | `s.lstrip('0')` 一行 | 手写 `while` 循环，注意保留至少一位 |
| 字符串比较 | `s == t` 或 `s <= t` | `strcmp(s, t) <= 0` |
| 字符串长度 | `len(s)` | `strlen(s)`，且每次调用都遍历到 `\0` |
| 镜像复制 | `s[:half][::-1]` 切片 | 手写 `for` 循环：`t[i] = t[len-1-i]` |
| 大数减 1 | `int(s[:half]) - 1` | 手动借位：`while(i>=0 && s[i]=='0') { s[i]='9'; i--; } s[i]--;` |
| 前导零判断 | `s[0] == '0'` 一样 | 一样，但要记得 `s[0]` 是 `char`，比较用 `'0'` 不是 `0` |

> **本质区别**：Python 的字符串是**对象**，切片、反转、比较都是封装好的操作。C 的字符串是**字符数组**，每个操作都要你亲手操纵指针/索引。

## 七、构造法的核心代码骨架（C）

```c
#include <stdio.h>
#include <string.h>

int main() {
    char s[205];
    scanf("%s", s);  // ✅ 用字符串读入，保留前导零，无视长度限制

    int n = strlen(s);

    // 1. 去前导零（保留至少一位）
    int st = 0;
    while (st < n - 1 && s[st] == '0') st++;
    // 把有效字符前移...（略）

    int m = n - st;  // 实际长度
    int half = (m + 1) / 2;

    // 2. 镜像构造候选回文
    char t[205];
    strcpy(t, s);
    for (int i = m / 2; i < m; i++)
        t[i] = t[m - 1 - i];

    // 3. 候选 ≤ 原数？直接输出
    if (strcmp(t, s) <= 0) {
        printf("%s\n", t);
        return 0;
    }

    // 4. 候选 > 原数，前半部分减 1（手动借位）
    int i = half - 1;
    while (i >= 0 && s[i] == '0') {
        s[i] = '9';
        i--;
    }
    if (i >= 0) s[i]--;

    // 5. 减 1 后出现前导零？输出 m-1 个 9
    if (s[0] == '0') {
        for (int i = 0; i < m - 1; i++) printf("9");
        printf("\n");
        return 0;
    }

    // 6. 重新镜像输出
    for (int i = m / 2; i < m; i++)
        s[i] = s[m - 1 - i];
    printf("%s\n", s);
    return 0;
}
```

## 八、给我的建议

1. **看到 "大整数" / "长度 ≤ 200" → 第一反应必须是字符串**，不能是 `int` / `long long`。
2. **Python 能写的逻辑，C 也能写**，只是 C 要求你显式管理内存和数组。先写 Python 验证算法，再逐行翻译成 C。
3. **回文数构造题 = 字符串镜像 + 大数减 1**，这是固定套路，记住这个模式。
4. **边界情况永远先测**：`10`, `100`, `1000`, `999`, `99` 这些 "整十整百" 的数最容易出 bug。

---

## 九、一句话总结

> **我的思路**："从 n 往下一个个试，找到第一个回文" —— 逻辑对，但只适用于小数字。  
> **正确思路**："直接把 n 的后半部分掰成前半部分的镜像，如果超了就把前半部分减 1 再掰" —— 字符串层面的几何操作，与数字大小无关。

**枚举是暴力，构造是艺术。**
